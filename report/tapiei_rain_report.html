領先時間分佈 (分)

通過 = 三項同時達標（領先中位數≥門檻、命中率≥門檻、FP≤門檻）。任何一項過不了 → 沒 edge，不做。

演算法假設（可調）：落雨事件 = 距分析點「雨綁定半徑」內、且與前一筆雨間隔≥N 分的第一筆雨；命中 = 事件前 N 分內有合格閃電；領先 = 事件時間 − 窗內最早合格閃電；FP = 一個觸發事件後 N 分內該點無落雨。觸發事件以 N 分為 debounce 合併。

E2 / E4為何不在這支工具裡

E2 · 雙偏極化雷達：格點 or 影像？

這是查 API catalog 的手動研究，不是寫 code 能代勞的。要做的事：

  • 到 opendata 平臺找樹林/林口雙偏極化降雨雷達資料集，看回傳是 JSON 格點數值 還是 PNG 圖磚。
  • 若為影像：估每 2 分鐘地理對位 + 逐 H3 取像素的 CF Worker CPU/頻寬，判斷 $0 是否還成立。
  • 分支：純影像且破免費額度 → 降級為標準雷達格點 F-B0046-001（QPESUMS 1hr QPF），但要重問降級後還有沒有 edge。

E4 · 推播端到端延遲

單一 HTML 測不了 mobile push 端到端——需要真實 push stack（Expo Push / APNs）、實機背景、跨網路狀況。Web Push 是另一條 stack 且 iOS 受限，量不到你會出貨的那條路徑。E4 留到 Phase 3 用真實 App 殼量。

提醒：E4 的端到端延遲若 > E3 量到的領先時間，領先就只是理論值——這條 kill criterion 一定要在寫商業邏輯前驗。
