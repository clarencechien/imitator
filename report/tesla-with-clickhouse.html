Tesla Comet：千兆級 (Quadrillion-Scale) 觀測平台架構與技術解析

基於 ClickHouse 構建的超大規模、完全兼容 PromQL 的自研監控數據平台。

1. 核心需求與原有技術瓶頸

Tesla 營運涵蓋超級工廠、能源系統與全球聯網車輛，其監控系統面臨極端的數據吞吐量與嚴苛的可用性要求。傳統的 Prometheus 架構無法滿足其規模。

維度 Tesla 系統要求 Prometheus 原生限制
擴展性 (Scalability) 需處理海量實時數據，支援無限水平擴展。 單節點設計，無法水平擴展，垂直擴展存在硬體天花板。
高可用性 (Availability) 監控數據遺失可能導致物理世界災難，容錯率為零。 單點故障 (SPOF) 風險高，節點停機即遺失當前指標。
生態兼容 (Compatibility) 必須 100% 支援 PromQL，無縫銜接現有儀表板與告警。 原生支援，為參考標準。
數據分析 (Queryability) 需支援數年資料的長期分析、預測與異常檢測。 DSL (PromQL) 功能受限，不利於複雜的商業邏輯與長期大數據分析。
技術決策： 選擇 ClickHouse 作為底層儲存。主因：卓越的分析效能、可水平擴展、高可用性、支援可執行的使用者自定義函數 (UDFs，這對實現 PromQL 轉換至關重要)。

2. Comet 雙軌架構解析

2.1 寫入路徑 (Ingest Pipeline)

圖 1: 數據寫入路徑

  • 採集層： 數以萬計的 OpenTelemetry Collectors 部署於基礎設施中，收集數據。
  • 緩衝層： 透過 OTLP 協議將指標發送至 Kafka (或相容佇列)，進行流量削峰。
  • 轉換層： 自研的 Comet ETL 進程從 Kafka 消費數據，將 OTLP 格式轉換為結構化的 ClickHouse INSERT 語句。
  • 寫入策略： 採用激進的批次處理 (Batching)，每批次達數百萬筆，最大化 ClickHouse 的寫入效能。

2.2 查詢路徑 (Query Pipeline)

圖 2: 數據查詢與轉譯路徑

  • 協議相容： Grafana 與告警系統 (Alerting) 透過標準 Prometheus API 發起查詢，系統完全無感底層替換。
  • 核心引擎 (Transpiler)： Comet 內建即時轉譯器，將 PromQL 動態轉換為 ClickHouse SQL 執行。
  • 格式封裝： ClickHouse 返回數據後，Comet 將其重新格式化為與 Prometheus API 完全一致的 JSON 結構返回前端。
  • 正確性驗證 (Comet Test Suite)： 部署獨立測試套件，同時向原生 Prometheus 與 Comet 發送相同的 PromQL，驗證返回結果需達到位元組級 (byte-for-byte) 100% 一致。

Tesla 團隊以 每秒 10 億列 (1 Billion rows/sec) 的速度連續寫入 11 天。最終成功攝入 超過一千兆列 (1 Quadrillion rows) 的數據。

4. 專家視角：盲點、風險與生存偏差評估

此篇技術案例為 ClickHouse 官方部落格之成功案例分享，具備一定程度的「生存偏差 (Survivorship Bias)」。作為理性的決策者，必須審視未被提及的工程代價：

風險一：Transpiler (語法轉譯器) 的隱性維護成本極高

盲點： 報告宣稱可即時將 PromQL 轉為 SQL，並透過千項測試保證位元組級一致。但 PromQL 的語法規則、函數庫在持續演進，某些基於時間序列向量的運算在關聯式/列式 SQL 中極難表達（這也是他們需要 UDF 的原因）。
風險： 企業將被迫綁定專職研發團隊，持續追蹤並維護這個「翻譯引擎」。這不是一次性開發，而是永無止境的技術債。

風險二：高基數 (High-Cardinality) 處理的架構細節缺失

盲點： 宣稱「完全不畏懼高基數問題」。然而 ClickHouse 本身雖強大，但面對海量標籤組合（Labels），若未妥善設計 `ORDER BY` 鍵值、跳躍索引 (Skip Indices) 或物化視圖 (Materialized Views)，查詢掃描量將呈指數爆炸。
風險： 盲目照搬架構將導致 ClickHouse 叢集在複雜查詢下發生 OOM (Out of Memory) 或 I/O 瓶頸。Tesla 必然投入了大量心力在 Schema 設計上，而這並未公開。

風險三：資料去重 (Deduplication) 與延遲寫入問題

盲點： 從 OTel 經 Kafka 寫入 ClickHouse，分散式系統中不可避免會產生重複數據。ClickHouse 原生的 `ReplacingMergeTree` 是非同步去重。
風險： 查詢結果的實時絕對精確性可能會受妥協，或者需要在轉譯後的 SQL 中強制加入複雜的 `argMax` 或 `LIMIT 1 BY` 邏輯，這會大幅消耗查詢時的 CPU 資源。

5. 決策建議與替代方案 (Action Items & Alternatives)

替代方案評估 (Alternatives)

除非具備 Tesla 級別的基礎設施研發能力與專有 SQL 分析需求，否則應優先考慮以下市占率高且專為此場景設計的開源方案：

  • ▶
    VictoriaMetrics:
    原生支援 PromQL 與高基數優化。資源消耗極低，為 Prometheus 擴充的最優性價比選擇。無需自研轉譯器。
  • ▶
    Grafana Mimir / Cortex:
    基於雲原生物件儲存 (S3/GCS)，天然解決海量儲存與長期保留問題。與 PromQL 生態系 100% 契合。
  • ▶
    ClickHouse 生態現成工具:
    若必須使用 ClickHouse，應優先評估開源專案如 qryn 或 promhouse，避免從頭打造 Transpiler 與 ETL。

執行步驟建議 (Action Items)

  • 需求量化： 定義企業目前的指標吞吐量 (Rows/sec)、基數規模與資料保留期。若未達每秒百萬寫入級別，直接採用 VictoriaMetrics。
  • 概念驗證 (PoC)： 若決定採用 ClickHouse，第一步必須對 ClickHouse 的 Schema (Table Engine) 進行高基數資料壓測。重點測試 `ORDER BY` 鍵的排列組合對查詢速度的影響。
  • 解耦架構： 效仿 Tesla 架構，確保寫入端導入 Kafka。這能確保在 ClickHouse 進行表結構變更或叢集重啟時，監控數據不會遺失。
  • SQL 能力評估： 盤點現有 SRE 團隊的能力。若團隊只熟練 PromQL，強制推廣底層為 ClickHouse 的系統將導致後續自定義報表開發受阻。需將「UDF開發」納入平台團隊的維護職責。
