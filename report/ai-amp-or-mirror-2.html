CLAUDE CODE 實用指南 · 針對 CS 工作者 · 2026.07

系列下篇 · 上篇:同一個工具,為什麼結果差這麼多?

工具在手,
怎麼用才算數?

上篇的結論是:生成趨近免費,價值集中到「鑑別力」,而鑑別力外包不掉。下篇回答接下來那個問題——那鑑別力要放在哪裡?答案:寫成 repo 裡可執行、可搬家的工件。以 Claude Code 為主軸,帶到 Cowork 與網頁版,整理你可能知道、也可能不知道、但必須知道的事。

核心原則:補模型不足的東西,下一版就被吃掉;編碼你約束的東西,模型越強、槓桿越大。所以——往「確定性」與「你擁有」那一象限搬。

TL;DR — 三十秒版本

上篇說鑑別力外包不掉;下篇說:那就把它寫成工件——放進 repo、做成閘門、保持可搬。

  1. 先給它一個能自己跑的檢查。官方 best practices 第一條:測試、build、可比對的輸出。這是「你得盯著」和「你可以走開」的差別——沒有檢查,你就是驗證迴路本身。
  2. CLAUDE.md 是路標不是倉庫。判準只有一句:「拿掉這行會不會犯錯?」不會就刪。知識放 repo 和 skills,CLAUDE.md 指路;要擋住的事交給 hook,因為 md 是建議、hook 是保證。
  3. auto memory 是在位者優勢製造機。它讓你用最久的工具看起來最好,跟模型品質無關;比較工具時必須冷測(關記憶)+ 熱測(開記憶)各跑一輪,對照著讀。
  4. Chat 是對話,Cowork 交辦檔案,Code 交辦 repo。再用兩個問題選 surface:這次跑在誰的 CPU 上?要不要人盯?CLI 給自動化,桌面給互動,網頁給長時任務。

01 · 先破除誤區

QA:可能知、可能不知、但必須知的八件事

延續上篇的體例:橘色是體感或傳說,綠色是文件或實證。這八題全部來自 CS 使用者實際會踩的地方。

「我沒寫 CLAUDE.md 也產出了好幾個 app,所以它不重要了?」

體感網頁版開空 repo、CLAUDE.md 留白,照樣出貨,所以那東西過譽了。

實情你測的是 greenfield——code 是模型自己寫的,慣例是它當場訂的,沒有「猜不到的東西」可寫,該放進 CLAUDE.md 的內容恰好是空集合。它付錢的三個條件是:既有的、有歷史的 codebase;跨 session 跨月份回頭改;以及不在程式碼裡的外部約束(裝置資源上限、離線假設、不可引入的相依)。玩具專案留白是對的;有歷史與約束的專案,空白才是錯的。順帶一提,補記憶那部分用途確實被 auto memory 吃掉了——縮小是真的,消失不是。

「CLAUDE.md 寫越詳細越好?」

傳說把 coding style、架構、流程全塞進去,模型就會全部遵守。

文件方向相反。官方判準只有一句:「拿掉這行會不會讓 Claude 犯錯?」不會就刪;檔案太長時重要規則會淹沒在雜訊裡,模型會忽略掉一半。該排除的是「Claude 讀 code 就能看出來的任何東西」。廣泛適用的少數規則留在 CLAUDE.md;偶爾才用到的領域知識放 skills 按需載入;每次都必須發生的事寫成 hook。一份值錢的 CLAUDE.md 在 2026 年長得像路標:二十行,指向 HANDOFF、ADR、skills 的位置,本身不裝知識。

「repo 裡有 README、handoff、一堆 md,誰記得都無所謂?」

體感我不記得、模型不記得,repo 記得,所以問題解決了。

實情「repo 記得」只解決三層裡的第一層。第一層是存在——資訊沒丟;第二層是會不會在對的時刻進到 context——CLAUDE.md 特殊之處不是內容,是保證每次載入;其他 md 要等模型決定去讀才進得來,而它不會全讀。skill 是中間態:有 description 當索引,像有目錄的書;散落的 md 是沒有目錄的書堆。第三層是擋不擋得住——所有 md(含 CLAUDE.md)都在建議層,只有 hook 在保證層。這正是上篇 context store 的論點落到工具層:意圖(spec)、行為(test)、結構(fitness function)三個驗證迴圈,都得是機器可執行的,不是散文。

「auto memory 開著,它越用越懂我,很好啊?」

體感跨 session 記得我的專案,這是純粹的加分。

實情方便是真的,但三個代價要知道。一、它是在位者優勢製造機:你用最久的工具知道得最多,表現最好,跟模型品質無關——你在比「共事三個月的同事」和「第一天上班的人」,而且偏誤會複利(越好用→越常用→記越多)。二、它會把一次性抱怨變成永久政策,而且是靜默的;定期跑 /memory 稽核它記了什麼,不是潔癖,是上篇「驗證」在記憶層的版本。三、它是廠商格式、搬不走;你寫下來的 CLAUDE.md、skills、hook 是純文字,可以整包搬去任何一家。累積可搬的資產,稽核搬不走的方便。

「御三家個性差很多,選對家就贏了?」

傳說某家適合寫 code、某家適合分析,跑分只是參考,體感才準。

實證個性差異可測量是真的:模型家族間的行為差異統計顯著、無提示也穩定出現,而且某些行為軸(如道德穩健性)變異超過一個數量級、幾乎完全由家族解釋——後訓練塑造的。但兩個修正:一、傳統跑分已飽和(60 個基準裡 29 個高飽和;445 篇基準論文只有 16% 做過統計檢定),頂端一兩分的差距是雜訊,「跑分差不大」這句話本身是對的。二、你體感到的差距,大半不是模型:只改 harness 不換模型,agent 排名能移動 20 名以上,再疊上 memory 的在位者優勢和你自己的適應。所以正確問法不是「誰強」,是「哪家的預設行為 × 工具鏈,配我的任務」——這是配適問題,用下一題的方法自己量。

「要比較工具,找個 benchmark 榜單看就好?」

做法榜單沒有你的任務分佈,而且記憶變數會污染你自己的日常體感。有效的做法一個下午做得完:從真實工作抽 10–20 個任務,冷熱各跑一輪。冷測(CLAUDE_CODE_DISABLE_AUTO_MEMORY=1、乾淨目錄、無 CLAUDE.md、字面相同的 prompt)回答「模型誰強」;熱測(記憶全開)回答「我該用哪套」。對照著讀才有資訊:冷熱都 A 贏→放心用;冷測平手熱測 A 贏→你的優勢是累積狀態,狀態可以搬;冷測 B 贏熱測 A 贏→你被鎖住了,先搬狀態再測一次。盲評結果,別看著名字打分。

你可能沒想到的:「hook 就是我的安全機制?」

體感寫了 PreToolUse 擋 rm -rf,安全了。

文件官方講得很直白:hook 的過濾是 best-effort、解析失敗時 fail open(放行),強制 allow/deny 要用 permission 系統,hook 管的是流程紀律。另外三個必踩的坑:exit code 1 是 Unix 失敗慣例,但 Claude Code 把它當非阻擋錯誤照做,要擋只有 exit 2;PostToolUse 擋不住任何事(工具已跑完);Stop hook 連擋 8 次後會被覆蓋直接結束——它是強力推一把,不是絕對閘門。分工記法:permission 管安全,hook 管紀律,CLAUDE.md 管建議。

你可能沒想到的:「80% 完成得飛快,所以整體變快了?」

體感鷹架長出來、能編譯、demo 會動,這專案穩了。

實情這是 context store 那篇最有價值的觀察:AI 讓前 80% 感覺飛快,而最後 20% 才是架構所在——整合、邊界情況、沒文件的效能限制、當年取捨的組織記憶;流程中最快的那段,遮住了最慢、後果最重的那段。再疊上上篇的鐵證:同批資深工程師體感 +20%、實測 −19%。兩件事指向同一個紀律:速度要用 merge lead time、返工率這類實測數字量,不要用 demo 會動的那一刻的感覺量。PR 範本裡強制寫「80/20 交接說明」(哪些是鷹架、哪些真的想過)是最便宜的對策。

02 · 核心圖像

兩個軸,決定每一份設定的命運

開頭那張四象限就是全篇的骨架,這裡把兩個軸講清楚。

軸一 — 強制力

建議性 vs 確定性

官方原話:CLAUDE.md 與 memory 都被當成 context,不是強制設定;要無論如何擋住某個動作,用 PreToolUse hook。建議層(md、skills)靠模型自覺;保證層(hooks、tests、permission)靠機器執行。上篇說鑑別力不能只放在你腦裡——在工具層的對應就是:不能只放在建議層。

軸二 — 持有權

廠商持有 vs 你持有

寫進 repo 的純文字(CLAUDE.md、skills、spec、tests、hooks)可以整包搬去任何一家、可以 diff、可以 review;auto memory 是廠商格式,綁定工具。上篇「補不足 vs 編碼約束」的判準在這裡多了一個理由:編碼約束的工件同時也是你的反鎖定保險。

推論 — 投資方向

往右上搬

右上(確定 × 你持有)是唯一「模型越強、價值越高」的一格:模型越能自主跑長任務,你的閘門與不變量被執行的次數就越多。左下(建議 × 廠商)最省力,但那是方便,不是資產。每次想「這條規則放哪」,答案幾乎永遠是:能寫成檢查就寫成檢查,能進 repo 就進 repo。

03 · SURFACE 分工

Chat、Cowork、Code:一句話 + 兩個問題

一句話記法:Chat 是對話,Cowork 是交辦檔案工作,Claude Code 是交辦 repo。之後只剩兩個問題:這次跑在誰的 CPU 上?要不要人盯著?

Surface交辦對象算力在哪獨有能力適合
Chat 問題與思考雲端 最快啟動、Memory 個人化 一次性問題、當思考對手、輕脈絡任務
Cowork 一般檔案與工作流隔離 VM 免 terminal 的 agentic 執行 整理資料、產文件、資料分析、排程雜務
Code · CLI repo(commit / PR)你的機器 唯一可被腳本呼叫:pipe、cron、CI 所有自動化;你既有的 claude -p 管線原封不動
Code · 桌面 repo你的機器 平行 session(git 隔離)、視覺化 diff、內建終端;Linux beta 已含 Cowork 互動式開發;要人盯著 review 的工作
Code · Web repo(回你 PR)Anthropic 託管 VM 零安裝、闔蓋照跑、多任務平行;Routines 雲端排程 長時間、吃資源、你要出門的任務;弱算力裝置的主力

三個備註:同一份 CLAUDE.md、settings、MCP 設定跨 surface 通用,切換沒有成本;桌面版取代不了 CLI(要 claude 指令還是得裝);真正只剩 Win/Mac 獨佔的是 Computer Use 與語音,其餘「桌面超能力」Linux beta 都有——Chromebook 的 Crostini(Debian bookworm)裝得起來。

04 · 帶著走

閘門階梯:從「盯著看」到「可以走開」

官方 best practices 幾乎所有建議源自一個約束:context window 是最重要的資源。而第一原則是——給 Claude 一個它能自己跑的檢查。沒有檢查,「看起來做完了」就是唯一訊號,你就是驗證迴路。閘門有四階,由弱到強:

G1
同 prompt 內迭代 — 在指令裡直接要求「跑測試,不過就修到過」。今天就能用,但靠模型自覺。
G2
/goal 條件 — 獨立評估器每輪重檢目標是否成立,跨 session 有效。官方說 G2、G3 是「無人值守能正確收尾」的關鍵。
G3
Stop hook 腳本 — 檢查寫成腳本,exit 2 擋住回合結束,stderr 餵回去當理由;記得處理 stop_hook_active 防迴圈。連擋 8 次會被放行,所以它是強力推一把,不是絕對閘門。
G4
驗證 subagent — 做事的不當評分的:讓另一個乾淨 context 的模型去反駁結果。無人值守的終點形態。

一週 easy path(照順序)

  1. /init 產生 CLAUDE.md,然後砍到 100 行以內——每行問「拿掉會不會出錯」,不會就刪。
  2. 挑一個你已經有的檢查(pytest、build、lint),定成規矩:每個任務結尾跑它並貼輸出當證據——官方明說看證據比自己重跑快,也比「宣稱成功」可信。
  3. 多檔案改動先 plan mode;同一問題糾正超過兩次就 /clear 重來——乾淨 session 配更好的 prompt,幾乎總是贏過一長串失敗嘗試。
  4. 把「絕不允許」的事從 CLAUDE.md 搬進 hook(PreToolUse 擋相依、Stop 跑不變量);「偶爾才用」的領域知識搬進 skills。
  5. 跑 /memory 稽核 auto memory 記了什麼;刪掉那些一次性抱怨變成的「政策」。
  6. 最後,寫你的領域 skill——那一百行 markdown(部署約束、不變量)是整條鏈上唯一不會被下一版模型吃掉的東西。

名詞備註 · 中英對照

建議層 / 保證層advisory / deterministic
CLAUDE.md、skills、memory 是建議——模型讀了但可以不理;hooks、tests、permission 是保證——機器執行,不問模型意見。
CLAUDE.md
每個 session 保證載入的那份 md。2026 年的正確用法是路標(指向知識所在),不是倉庫(裝知識本身)。
skillagent skill
帶索引(description)的領域知識包,模型按需載入。相對於散落 md 的差別:有目錄的書 vs 書堆。
hook
在生命週期特定時點自動執行的 shell 指令。stdin 吃 JSON、exit 2 阻擋並把 stderr 餵回模型;exit 1 不會擋(最大的坑)。
auto memory
Claude 自己記的跨 session 筆記(build 指令、你的糾正),預設開啟,存在 ~/.claude 下、廠商格式。方便,但要用 /memory 稽核。
session memory
對話層級的背景摘要,讓 /compact 即時、跨 session 接得上。跟 auto memory 是兩層不同的東西。
fitness functionarchitectural fitness function
把架構規則寫成可自動執行的檢查(如「不得引入某相依」「延遲不得超標」),進 CI 擋合併。context store 三迴圈之一。
80/20 交接80/20 handoff
AI 讓前 80% 飛快、最後 20% 才是架構所在。PR 裡註明哪些是鷹架、哪些真的想過,就是把這條裂縫變成可見的。
冷測 / 熱測cold / hot eval
關記憶、乾淨環境測「模型誰強」;開記憶測「我該用哪套」。兩者對照才能分辨優勢來自模型還是累積狀態。
在位者優勢incumbent advantage
用越久的工具記得越多、表現越好、你越不想換——與模型品質無關的自我強化迴圈。
