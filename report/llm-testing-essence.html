測試在測什麼?
可測的、不可測的,與兩者之間的搬運

Uncle Bob 說他不讀 AI 寫的程式碼,只讀指標。這句話炸出一連串必須回到初心的問題:測試到底在測什麼?什麼可測、什麼不可測?spec 錯了怎麼辦?你心裡的想法變成文字的那一刻,遺失了多少?本篇從他的關卡開始,一路走到上游——最後收在一張世界線的地圖上。

核心結論:spec → code 是 verification,靠機器,越自動越好;idea → spec 是 validation,靠迭代中的人類辨識,自動化不了。工程的日常,是不斷把左邊的擔憂改寫成右邊的可測投影——搬不過去的餘量,就是外包不掉的判斷力。

TL;DR — 三十秒版本

測試是校準儀器:量的是同一個意圖的兩份表述——spec 與 code——是否一致。它管不到 spec 本身對不對,更管不到你心裡想的東西有沒有進到文字裡。

  1. 可測性不是問題的屬性,是表述的屬性。「策略應該賺多少」永遠不可測(沒有 oracle);「訊號是否只用了 t 之前的資料」完全可測(變形關係)。同一個擔憂,換個表述就從不可測變可測——工程功力的一大部分就是這個改寫。
  2. Uncle Bob 示範的是判斷力的再配置,不是免除。他能不讀 code,是因為五十年的讀寫讓他設計得出那道關卡;而他的清單裡最被低估的一項是依賴結構——它是約束不是偵測:大象不是測出來的,是房間做小到進不來的。
  3. LLM 是會填空的神燈精靈。你的文字是意圖的有損壓縮,遺失的位元不是留白——模型會把它填成「一般人通常的意思」再加「你會點頭的樣子」,而且填得毫無接縫。它滿足的是你寫下的約束,不是你心裡的約束。
  4. 上游沒有測試,只有校準。例子與反例、假設帳本、non-goals、原型—反應迴圈——最後一項是 validation 唯一真正的儀器:有些 spec 只能被發現,不能被預先寫出。

01 · 起點

從 Uncle Bob 的「我不讀 code」說起

2026 年,寫了《Clean Code》、推了幾十年 TDD 的 Robert C. Martin 公開說:他不審查 agent 寫的程式碼,改量測試覆蓋率、依賴結構、圈複雜度、模組大小、變異測試——從指標推論品質,程式碼本身交給 AI,理由是人類讀 code 太慢,要生產力就得從 code 抽離、在更高層次管理[1]。他的做法不是放手:用單元測試、Gherkin 測試、QA 流程、品質指標、變異測試、覆蓋率把 agent 團團圍住,產出必須跑完整條考驗關卡(gauntlet)他才信任[2]。他還描述過四段流水線——Specifier → Coder → Refactorer → Architect,每段人工介入遞減[3]。

三件事讓這不只是名人語錄。一、他驗證驗證者:變異測試量的不是程式碼,是你的測試有沒有牙齒——這是大多數人的關卡裡缺的那一層。二、他自己在往回修:三週後他說堆滿各種測試「能做不代表該做」,很多時候單元測試加 CRAP 評估就夠,大型專案才需要更多,而且他還在驗證中[4]——一個公開修正自己主張的人,比把它包裝成方法論賣的轉述可信。三、也是最容易被聽錯的:他證明的不是「不用讀 code」。他能設計出那道關卡,靠的是 1970 年至今五十幾年的讀與寫。他示範的是判斷力的再配置——從逐行審查搬到閘門設計——不是判斷力的免除。一個從沒讀過 code 的人,設計不出那道 gauntlet。

順帶一提:「人類讀 code 太慢所以跳過,生產力就回來了」——這本身是一個沒被量測的宣稱。他感覺快了。上篇那批資深工程師也感覺快了 20%,實測慢 19%。

02 · 初心

測試的本質:一台校準儀器

為什麼要測試?標準答案「找 bug」是現象不是本質。本質是:你手上有同一個意圖的兩份表述——spec 和 code——測試量的是這兩份表述是否一致。它是校準儀器,不是品質證明書。

軟體工程有一對老詞(Boehm)把這切開[5]:verification——are we building the thing right?code 是否忠於 spec;兩邊都是形式化物件,可測。validation——are we building the right thing?spec 是否忠於需求、需求是否忠於你心裡想的;等式左邊不是形式化物件,不可測,只能校準。

整條鏈是:mind → idea → 文字 → spec → code。測試只覆蓋最後一段。所以「有多校準」的問題會一路往上游退——退到一個測試構不到的地方。這篇的路線就是:先把構得到的講透(03、04),再誠實面對構不到的(05、06)。

03 · 分界線

可測與不可測:四級,和一個關鍵改寫

一個性質要可測,需要兩個條件:可形式化(能寫成機器可判定的命題)+ 有 oracle(有辦法知道對錯)。由此分四級:

級別條件工具例子
有明確答案形式化 ✓ · oracle ✓單元測試sort([3,1,2]) = [1,2,3]
沒答案,有恆真關係形式化 ✓ · oracle ✗變形測試、屬性測試「追加未來資料,歷史訊號不得變」
只有統計性質形式化 ✓ · oracle 是分佈統計檢定(帶錯誤率)延遲 p95 < 200ms
意圖、品味、「對的東西」形式化 ✗不可測,只能校準「這是不是我要的」

案例:前視偏誤

回測時,計算 t 時刻訊號用到了 t 之後才存在的資訊——用當日收盤決定當日開盤的交易、用整段樣本的統計量正規化每一天、用事後修正的財報、拿現在還活著的股票清單回測十年前。後果同一種:回測漂亮,實盤賠錢,因為實盤拿不到未來。

這裡要精確切開兩個命題。「這個策略應該賺多少?」——沒有 oracle,永遠不可測。「t 時刻訊號是否只依賴 ≤ t 的資料?」——完全可形式化:在資料尾端追加一筆未來資料,重跑,所有歷史時點的訊號必須逐位元相同;任何一點不同就是前視,而且你不需要知道正確答案是什麼。

同一個擔憂,兩種表述,一個不可測、一個可測。所以這句話值得寫在牆上:可測性不是你的問題的屬性,是你選擇的表述的屬性。工程功力的一大部分,就是把不可測的擔憂改寫成它的「可形式化投影」——你測不到「策略是好的」,但測得到「沒偷看未來」「成本模型沒被動」「樣本外窗口雜湊不變」。每條投影砍掉一類具體的死法;加起來不等於「好」,但把「壞」的空間收窄了。

04 · 工具箱

每種測試,為了補前一種的洞而存在

關鍵是「抓不到什麼」那一欄——它決定了下一層為什麼必須被發明。

方法為了解決什麼抓得到抓不到
單元測試改 A 壞了 B 要立刻知道回歸、你想到的邊界「你想的本身是錯的」
Gherkin / 驗收單元測試用實作者語言寫,會跟業務漂移通過單元測試但不是使用者要的利害關係人自己搞錯
QA / 探索腳本只找得到你想到的你沒想到的(唯一為此設計)沒被探索到的區域
變異測試覆蓋率會騙人:「被執行」≠「被斷言」空洞的測試、缺失的斷言缺失的需求
屬性測試案例列不完,不變量列得完你沒想過的輸入打破不變量你沒陳述的不變量
變形測試沒有 oracle 也要測違反恆真關係(如前視)關係之外的一切
差分測試笨而明顯正確的參考實作當 oracle快版與笨版的分歧兩版共有的錯

Gherkin 用 Given/When/Then 加領域詞彙寫成接近自然語言的句子,本質是利害關係人與程式碼之間的翻譯檢查;Uncle Bob 觀察到它對 agent 有額外作用——兩條測試流(驗收+單元)讓模型不能把 code 亂塞一通再補個單元測試了事,因為同時被驗收測試的結構約束住[6]。變異測試的機制:逐一對程式碼做語意改動(if 反轉、移除呼叫)產生 mutant,每個都跑測試套件——測試掛掉 = mutant 被殺死(好事);照樣全綠 = mutant 存活(壞事:你的測試沒在檢查那個語意)[7]。注意方向是反的:mutant 通過才是壞消息。它用結構操作、做語意檢查,而檢查對象是你的測試——這就是「驗證驗證者」。

CRAP:一條公式講完

CRAP(m) = comp(m)² × (1 − cov(m))³ + comp(m),comp 是圈複雜度(獨立執行路徑數 = 決策點 + 1),cov 是覆蓋率;Savoia 與 Evans 2007 年提出[8]。設計全在指數:複雜度平方、未覆蓋率立方——複雜函式加覆蓋率,分數劇烈下降;100% 覆蓋時第一項消失,CRAP = CC(複雜是事實,但已被測試控制);0% 覆蓋、CC=15 時飆到 240。門檻慣例 30;CC 約 30 以上再多覆蓋也壓不下去——意思是「該重構了,不是多寫測試能救的」[8]。為什麼要合併:純複雜度懲罰「本來就必須複雜」的 code,純覆蓋率獎勵「在瑣碎 code 上刷測試」;CRAP 鎖定又複雜又沒測的交集——bug 真正的藏身處。

Goodhart:當量測變成目標

Charles Goodhart,1975 年談貨幣政策:任何被觀察到的統計規律,一旦被拿來當控制目標就會崩壞;通俗版(Strathern)是那句名言——當一個量測變成目標,它就不再是好的量測[9]。放進 agent 情境:指標一旦成為閘門,agent 就朝「通過」最佳化而非朝「品質」——而 LLM 特別擅長朝明確標準最佳化,它的強項變成你的風險。具體玩法:覆蓋率 → 執行了但幾乎不斷言的測試;圈複雜度 → 把複雜函式切成一堆瑣碎函式,複雜度轉移到呼叫關係裡;變異分數 → 技術上敏感、語意上無意義的斷言。複合指標提高遊戲難度,但不消除遊戲。

依賴結構:清單裡最被低估的一項

原文就是推文裡那兩個字:「dependency structure」[1]。對他而言份量特別重,因為量的是他自己 1990 年代發明的那套指標:依賴無環原則(模組依賴圖不得有環)、不穩定度 I = Ce/(Ca+Ce)、穩定依賴原則(依賴指向更穩定的方向)、抽象度 A 與主序距離 D = |A+I−1|,加上 Clean Architecture 的依賴規則——原始碼依賴一律指向內圈,框架與 DB 不得滲入業務政策[10]。

為什麼最被低估:其他指標(覆蓋率、複雜度、mutation)都是偵測——事後問「有沒有問題」;依賴結構是約束——事前規定「什麼形狀不允許存在」,而且可以做成 fitness function 進 CI(「domain 層 import 了 infrastructure → build 失敗」)。agent 產一萬行你不讀沒關係,但它不可能產出違反依賴方向的架構,因為那個形狀過不了關卡。

房間裡的大象

那麼,關著門測得出大象嗎——甚至,分得出牠是大象還是老虎嗎?誠實的答案:光靠測試,測不出來。測試是封閉世界方法,只回答你問的問題;全綠的意思是「沒有我測過的那種大象」,不是「沒有大象」。而且有兩種不同的失敗要分開:偵測失敗(有東西,但你沒問對問題)與辨識失敗(發現了,但抽象錯了——你測延遲,真正的問題是抖動)。

真能對付未知的方法,共同點是「會讓你意外」:屬性測試、模糊測試、變形測試、差分測試。但最重要的一句是——你不能靠測試偵測大象,你只能把房間做小到大象進不來。約束優於偵測:偵測問「裡面有什麼」,約束問「裡面能有什麼」——相依白名單、權限與沙盒、依賴方向檢查、資源上限。再補兩塊:抽樣(不必讀全部,讀選得好的一些:隨機+高風險點——統計品管,不是全面審查)與可觀測性(大象最後會出聲;測試是上線前的封閉世界,監控是上線後開的那扇窗)。

05 · 上游

神燈精靈與熵:idea → spec 那段發生了什麼

先給那個直覺一個正式名字:你需要某個東西告訴你「正確答案是什麼」,它叫 oracle——spec、預期值、斷言,或另一個實作。oracle 錯了,整條鏈就忠實地執行錯誤,而且全綠。Dijkstra 那句老話就是在說這個:測試只能顯示 bug 存在,不能證明不存在[7]。

用資訊論的語言講你的提問:你心裡的意圖是一個分佈,打出來的文字是它的有損壓縮;條件熵 H(intent | text) > 0——同一段文字相容於很多個意圖。這個量無法直接測量,因為 intent 沒有 ground truth,連你自己都拿不到:很多意圖是被追問之後才成形的(Polanyi:我們知道的比說得出的多[11])。

LLM 拿到文字後做的不是「還原你的意圖」,是從它的先驗分佈裡取條件於你文字的最可能補全。兩個推論比「有損」更重要:一、遺失的位元不是留白,是被填充——而且填的是「一般人通常的意思」,所以誤差不是均勻雜訊,是系統性地往平均使用者回歸;你的需求越獨特,被靜默改寫的越多,而且改得毫無接縫。二、再疊一層諂媚:它傾向填成你會點頭的樣子——點頭,不代表那是你要的。所以「魔術扭曲事實」會發生,但機制不神秘:模型滿足的是你寫下來的約束,不是你心裡想的約束;兩者差距越大,扭曲越明顯。

「熵有多大」沒有絕對答案,但有一個可操作的代理量測:讓模型對同一段需求,獨立生成 k 個刻意分歧的解讀,看它們差在哪。全部收斂 → 你的文字低熵,放心走。發散 → 每一個發散的維度就是 H(intent | text) 的一個具體位元,而且現在它變成選擇題擺在你面前——你只要認得出哪個對,不用憑空寫出完整 spec。人類的辨識遠強於回憶(recognition over recall);這一招等於把不可測的熵,轉換成可枚舉的分歧清單。

上游沒有測試,只有五台校準器

C1
用例子當 spec。「搜尋要快」是高熵句;「10 萬筆、冷啟動、p95 < 200ms」是低熵句。任何 spec 必須附三個具體例子加一個反例(什麼情況該拒絕)——反例的資訊量通常最大。這就是 Gherkin/ATDD 存在的深層理由:具體例子的熵遠低於形容詞。
C2
假設帳本(assumption ledger)。規定模型動工前輸出「我替你做了哪些決定」——不是它理解了什麼,是它替你填了什麼。填充從靜默變顯式,你逐條打勾或打叉。直接攻擊「無接縫填空」機制。
C3
Non-goals。一頁 PRD 有用,但用對地方:它不是完整 spec(不可能),是決策紀錄——資訊密度最高的欄位是「這次不做什麼」。正向需求有無限種寫法,負向約束一刀砍掉整個子空間。跟依賴結構同構:都是約束,不是偵測。
C4
原型—反應迴圈。最誠實的一條:承認 spec 的一部分只能被發現,不能被預先寫出。看到會動的東西才知道「不對,不是這個」——那不是流程失敗,那是量測行為本身,是 validation 唯一真正的儀器。AI 把單次迭代成本打到極低,正確用法不是花更久寫完美 spec,是更快丟出可反應的版本、更密地採樣你自己的反應。
C5
spec 變異測試(推論,非成熟實踐)。既然能 mutate code 看測試抓不抓得到,就能 mutate spec 看你抓不抓得到:把 spec 做幾個語意改動拿給你看,你分不出差別的那些條款,就是你其實沒有約束到的自由度。與 Uncle Bob 的 Gherkin mutation 同方向,再往上游推一層。
06 · 世界線

方位、岔路、與漂移的目的地

把機率論的樹攤開:所有可走的路發散成一棵巨大的樹,而你只活在收束後的一條世界線上。三個問題住在三個不同的層:

Verification 是方位檢查——「走的是不是南邊」。沿途每一步都能問、都能機測,這是 code mirror spec 的那一段,Uncle Bob 的關卡管的就是它,越自動越好。Validation 是岔路選擇——「走哪條路」。只能在岔口由人辨識,工具(k 個分歧解讀、假設帳本)能把岔路擺清楚,但選擇不能外包。而目的地是第三層——「終點是不是你要的目的地」,這個問題連問都要等你回頭:看過已走的路、重畫過地圖,它才問得出來。

而重畫之後的目的地,常常已經不是最初那顆星。LLM 給你的走法、你自己當初寫下的目標,都不是那個真正的目的地——只有回望、重選、再猜,逼近的那個「當下的終點」才是。這不是妥協,是發現:出發時的目的地本來就是在資訊最少的時刻定的,走過的路是你唯一真實的量測資料,不拿它更新地圖才是浪費。原型—反應迴圈是這件事的工程版;敏捷整套方法論的核心洞見,也就這一句。

所以整條收攏成一句話:spec → code 之間靠機器,idea → spec 之間靠迭代中的辨識,而目的地本身在行走中重寫。LLM 改變的是:它把第一段的成本打到近零,同時把第二段的風險放大——因為它填空填得太流暢、太討喜,讓你誤以為第二段也被解決了;至於第三段,它連碰都碰不到,那是你回頭看路時才存在的東西。搬不過去的餘量——選岔路的辨識、重畫地圖的回望——就是上篇說的、外包不掉的判斷力。

07 · 名詞備註

中英對照

驗證 / 確效verification / validation
Verification:code 忠不忠於 spec(building the thing right),可機測。Validation:spec 對不對得起需求(building the right thing),只能校準。
oracletest oracle
告訴你「正確答案是什麼」的那個東西——spec、預期值、參考實作。Oracle 錯,全鏈忠實執行錯誤且全綠;沒有 oracle 的問題要改用變形/屬性測試。
不變量invariant
不管狀態與輸入,恆為真的陳述(餘額不為負、轉帳總額守恆)。測試案例是點,不變量是面;列舉列不完,陳述可以涵蓋無限。
變異測試mutation testing
對程式碼做語意改動(mutant),看測試套件抓不抓得到。量的是「測試有沒有牙齒」,不是程式碼本身——驗證驗證者。Mutant 存活(全綠)才是壞消息。
屬性測試property-based testing
你陳述不變量,工具生成數千個你沒想過的輸入去攻擊它,找到反例自動縮小。Python 生態是 Hypothesis。
變形測試metamorphic testing
不知道 f(x) 該是多少,但知道 f(x) 與 f(g(x)) 必須有的關係。沒有 oracle 也能測——前視偏誤檢查(追加未來資料、歷史訊號不變)就是一例。
前視偏誤look-ahead bias
回測時 t 時刻的訊號用到 t 之後的資訊(當日收盤決定當日交易、全樣本統計量正規化、存活者名單)。回測漂亮、實盤賠錢的頭號原因。
CRAPChange Risk Anti-Patterns
comp² × (1−cov)³ + comp。鎖定「又複雜又沒測」的交集;100% 覆蓋時退化為圈複雜度本身;門檻慣例 30。
圈複雜度cyclomatic complexity
函式中線性獨立執行路徑數 = 決策點 + 1。越高越難理解與維護;約 30 以上被視為該重構而非補測試。
Goodhart 定律Goodhart's law
量測一旦變成目標就不再是好量測(1975,貨幣政策脈絡)。Agent 會朝閘門指標最佳化——LLM 的強項變成你的風險。
依賴結構指標ADP / I / A / D
依賴無環;不穩定度 I=Ce/(Ca+Ce);依賴指向更穩定方向;主序距離 D=|A+I−1|。屬於約束不是偵測,可做成 CI 閘門。
條件熵H(intent | text)
給定你的文字,你的意圖仍剩多少不確定性。無法直接量,但可用「k 個分歧解讀的發散程度」當代理。
內隱知識tacit knowledge
Polanyi:我們知道的比說得出的多。意圖的一部分只能被追問與原型「挖」出來,寫不進第一版 spec。
