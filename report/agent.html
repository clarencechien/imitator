A. AI 代理人核心概念

理解「什麼是 AI 代理人」以及「為何它們如此重要」是學習 AI 代理工程的起點。

AI 代理人（AI Agent）是一種能夠在少量（甚至無需）人類干預下，自主完成任務、做出決策並與環境互動的智慧程式。其核心特徵包括：

  • 自主性 (Autonomy)：能夠獨立運作。
  • 感知能力 (Perception)：能感知環境變化。
  • 決策能力 (Decision-making)：能根據目標和感知到的信息做出判斷。
  • 行動能力 (Action)：能執行任務。
  • 通常具備學習能力 (Learning)和推理能力 (Reasoning)。

可以將 AI 代理人想像成一位能夠「觀察、思考、行動」的數位助理。

AI 代理人的類型（舉例）：

  • 專家型 AI (Expert Agents)：專注於解決特定領域問題。
  • 互動型 AI (Interactive Agents)：強調與使用者進行自然語言互動。
  • 學習型 AI (Learning Agents)：能夠從經驗或數據中不斷學習。

應用場景：

從個人助理、企業客服到專業領域的醫療 AI 等，應用極為廣泛。

面臨的挑戰：

處理複雜工作流程的穩定性、運算資源需求、AI 的「幻覺」(hallucination) 現象等。

C. Jupyter Notebook 操作指南

`ed-donner/agents` 專案大量使用 Jupyter Notebook (約 92.8% 的內容)。熟悉其操作是學習本課程的必要前提。

Jupyter Notebook 是一個互動式計算環境，整合程式碼、文本、數學公式、視覺化圖表於一體（.ipynb 檔案）。

主要介面元件：

  • 筆記本名稱 (Notebook Name)：顯示在頂部，可修改。
  • 選單列 (Menu Bar)：檔案、編輯、檢視、插入、核心等功能。
  • 工具列 (Toolbar)：常用操作的快捷圖示。
  • 單元格 (Cell)：基本內容單位，主要類型有：
    • 程式碼單元格 (Code Cell)：編寫和執行程式碼。
    • Markdown 單元格 (Markdown Cell)：編寫格式化文本、插入圖片、連結、LaTeX 公式。
    • 原始單元格 (Raw NBConvert Cell)：內容在轉換時保持原樣。

常用操作與快捷鍵：

  • 執行單元格：Shift + Enter (執行並選取/建立下一個) 或 Ctrl + Enter (執行目前單元格)。
  • Markdown 語法：用於格式化文本，如 # 標題, **粗體**, *斜體*, - 列表。
  • LaTeX 數學公式：行內公式用 $...$，獨立公式塊用 $$...$$。
  • Esc：進入命令模式 (Command Mode) - 可移動、複製、刪除單元格。
    • A：在上方插入單元格。
    • B：在下方插入單元格。
    • DD (按兩次D)：刪除目前單元格。
    • M：將目前單元格轉換為 Markdown 單元格。
    • Y：將目前單元格轉換為程式碼單元格。
  • Enter：進入編輯模式 (Edit Mode) - 可編輯目前單元格內容。

建議您開啟 Jupyter Notebook，實際操作以上指令以加深理解。

D. API 金鑰與環境設定詳解

現代 AI 模型（尤其是 LLM）通常需要透過 API 存取。API 金鑰 (API Key) 是使用這些服務的「鑰匙」，涉及費用和安全問題，初學者需特別注意。`ed-donner/agents` 課程大量使用 OpenAI 模型。

什麼是 API 與 API 金鑰？

API (Application Programming Interface) 是軟體模組間通訊的機制。API 金鑰是一串用於控制 API 存取的字串，主要用於識別和驗證發出請求的應用程式或專案。

如何取得 OpenAI API 金鑰？

  1. 前往 OpenAI 官方網站註冊帳戶。
  2. 設定付款方式（API 使用通常收費）。
  3. 在其帳戶的 API Keys 管理頁面創建新的私鑰 (secret key)。
  4. 妥善保管此金鑰，切勿洩漏。

API Token 的概念與計費：

OpenAI API 以「Token」數量計費。Token 可理解為文字片段，輸入的提示 (Prompt) 和模型生成的回答都會消耗 Token。內容越長，Token 越多，費用越高。

環境變數與安全管理：

絕對不能將 API 金鑰直接寫在程式碼中或提交到公開的程式碼倉庫 (如 GitHub)。

最佳實踐是將 API 金鑰儲存在環境變數中。具體設定方法請參考 `ed-donner/agents` 專案 `setup` 資料夾中針對您作業系統的指南。

表 2：API 金鑰使用最佳實踐檢查表

檢查項目 是/否 (自我檢查)
我是否已將 API 金鑰儲存在環境變數中，而非直接寫在程式碼裡？
我是否了解此 API 的計費方式與 Token 消耗規則？
我是否已在 OpenAI 平台設定了適當的用量限制或預算提醒（如果適用）？
我是否已將包含 API 金鑰的設定檔（如 .env）加入到 .gitignore 檔案中？
我是否會定期檢視 API 金鑰的使用情況並在不再需要時將其停用或刪除？

負責任地管理 API 金鑰是進行 AI 開發的基礎。

E. Git 與 GitHub 專案協作基礎

`ed-donner/agents` 課程圍繞 GitHub 倉庫展開，因此學習者需具備基本的 Git 和 GitHub 操作知識，以便獲取課程程式碼、理解專案結構、追蹤更新。

版本控制系統 (VCS) 簡介：

VCS 記錄檔案變更歷史，可回溯特定版本，對軟體開發至關重要，支援多人協作、追蹤錯誤、管理版本。

Git 的基本概念：

  • 儲存庫 (Repository, Repo)：專案容器，含所有檔案及修訂歷史。
  • 提交 (Commit)：將檔案變更儲存到 Git 歷史的快照，有唯一 ID 和描述。
  • 分支 (Branch)：獨立的開發線路，可在不影響主線 (通常是 main 或 master) 的情況下進行實驗。
  • 合併 (Merge)：將一個分支的變更整合到另一分支。

GitHub 的基本概念：

  • 遠端儲存庫 (Remote Repository)：儲存在網路伺服器 (如 GitHub) 上的儲存庫，允許多人共享同步。
  • 派生 (Fork)：在 GitHub 上複製他人儲存庫到自己帳戶下，以便自由修改。
  • 拉取請求 (Pull Request, PR)：希望將修改貢獻回原始專案時，向維護者發起的審查與合併請求。

對本課程重要的操作：

最重要的是能夠將 `ed-donner/agents` 專案從 GitHub 複製 (git clone) 到本地電腦。例如：


git clone https://github.com/ed-donner/agents.git

學會查看提交歷史 (git log) 也有助於了解專案演進。

雖然初學者可能不會立即進行複雜的分支管理，但基本理解有助於更好地利用 GitHub 資源。

F. `uv` 套件管理工具介紹

`ed-donner/agents` 專案 README 提及使用 uv 安裝套件 (如 CrewAI)，表明 uv 是課程推薦或必需的環境設定工具之一。

什麼是 `uv`？

uv 是一個用 Rust 語言編寫的高性能 Python 套件安裝器和解析器。目標是作為傳統工具 (如 pip, pip-tools, virtualenv) 的直接替代品，提供更快速度和更簡潔體驗。

uv 如何取代傳統工具？

  • 取代 pip 和 pip-tools：
    • uv pip install：類似 pip install。
    • uv pip compile：類似 pip-compile，生成鎖定的 requirements.txt。
    • uv pip sync：類似 pip-sync，根據 requirements.txt 同步虛擬環境。
  • 取代 virtualenv：
    • uv venv：創建 Python 虛擬環境，速度更快。

uv 的主要功能與優勢：

  • 高性能：通常比 pip 快得多。
  • 簡化的工具鏈：單一靜態二進制檔案，整合多功能。
  • 廣泛的支援：支援可編輯安裝、Git/URL/本地依賴等。
  • 易於採用：uv pip 介面與 pip 相似。

在 `ed-donner/agents` 專案中，您可能會遇到如 uv tool install crewai 的指令。專案中也可能包含 uv.lock 檔案 (用於鎖定依賴版本)。

本模組將提供 uv 的安裝指南，並解釋其在課程環境設定中的作用和基本命令。其速度和易用性是推薦它的主要原因。

  • 訊息角色 (Message Roles)：system (AI 整體行為), user (使用者輸入), assistant (AI 回應)。
  • 函數呼叫 (Function Calling) / 工具使用 (Tool Use)：允許 LLM 請求呼叫外部函數/工具。
  • 基本代理迴圈 (Basic Agent Loop)：接收輸入 -> 建構提示 -> 呼叫 LLM -> 解析回應 -> (可選)工具呼叫 -> 生成最終回應。
  1. 用 OpenAI API 建構簡單問答機器人。
  2. 讓 AI 代理扮演「程式碼解釋器」。
  3. (進階) 定義新工具並讓代理呼叫。

C. 第三週：CrewAI

本週從單一代理擴展到多代理協作系統，學習使用 CrewAI 框架設計和建構由多個專職 AI 代理組成的團隊。

學習使用 CrewAI 框架設計和建構多代理團隊。重點：理解 CrewAI 核心組件 (Agent, Task, Tool, Crew) 的定義與互動。對應 GitHub 資料夾：3_crewAI (注意：原報告為3_crew，GitHub實際為3_crewAI)。

CrewAI 將複雜問題分解，指派給具特定專長和目標的自主代理人，組成「團隊」解決問題。

表 3：CrewAI 核心組件

組件 (Component) 描述 (Description) 重要參數 (Key Parameters)
代理 (Agent) 具有特定角色和目標的自主執行單元 role, goal, backstory, tools, llm, allow_delegation, verbose
任務 (Task) 需要完成的具體工作描述與預期輸出 description, expected_output, agent, context, tools
工具 (Tool) 代理執行任務時可使用的外部功能或資源 (取決於具體工具的定義)
團隊 (Crew) 由代理和任務組成的協作單元，負責協調執行 agents, tasks, process, manager_llm

流程 (Process)：支援順序流程 (Sequential) 和層級流程 (Hierarchical)。

  1. 修改 CrewAI 範例，增加新 Agent 角色和 Task。
  2. 為 Agent 添加新工具並修改 Task 以利用它。
  3. 嘗試將順序執行 Crew 改為層級流程。
  • Windows 環境下的 Unicode 編碼錯誤。
  • 工具未能正確載入或執行。
  • uv tool install crewai 或 uv tool upgrade crewai 命令執行問題。

D. 第四週：LangGraph

LangGraph 將複雜代理工作流程視覺化並建構成圖 (Graph)，適用於表示包含循環、條件分支和持久化狀態的代理行為。

掌握使用 LangGraph 建構有狀態、多步驟的 AI 代理應用。重點：理解核心概念 (State, Node, Edge)、圖的編譯與執行。對應 GitHub 資料夾：4_langgraph。

  • 狀態 (State)：Python TypedDict，追蹤圖執行過程中的數據。
  • 節點 (Node)：圖中的工作單元 (Python 函數/可呼叫物件)，接收狀態、返回狀態更新。
  • 邊 (Edge)：連接節點，定義控制流。分常規邊和條件邊。
  • 圖的編譯 (Graph Compilation)：compile() 方法將 StateGraph 編譯成可執行的 CompiledGraph。
  • 檢查點 (Checkpoints) 與記憶 (Memory)：持久化儲存圖的狀態，實現跨會話記憶。
  • 人機互動 (Human-in-the-Loop)：支援在流程中引入人工干預。
  • 工具節點 (ToolNode)：方便整合 LangChain 工具。
  1. 修改 LangGraph 聊天機器人範例，增加 user_name 狀態並互動。
  2. 為 LangGraph 代理添加新工具並修改條件邊邏輯。
  3. 檢視帶記憶功能範例的歷史狀態快照。

E. 第五週：AutoGen

AutoGen 強調透過自動化的多代理對話來解決複雜問題，模擬不同角色代理間的自然語言互動與協作。

了解並使用 Microsoft AutoGen 框架建立和協調多個 AI 代理。重點：理解不同類型代理 (UserProxyAgent, AssistantAgent)、配置行為、發起和管理對話。對應 GitHub 資料夾：5_autogen。

  • 可對話代理 (Conversable Agents)：能發送接收訊息，根據對話歷史和配置行動。
  • AssistantAgent：LLM 驅動的 AI 代理，可配置系統訊息定義角色、能力。
  • UserProxyAgent：代表使用者或其他代理互動，可獲取使用者輸入、執行程式碼、請求人類輸入。
  • 群組聊天 (Group Chat)：支援多代理參與，可設 GroupChatManager 管理流程。
  • 工具使用/函數呼叫：代理可使用外部工具或函數。
  • 程式碼執行：代理 (尤其 UserProxyAgent) 可直接執行 LLM 生成的程式碼。
  1. 修改 AutoGen 範例，改變 AssistantAgent 系統訊息扮演新角色。
  2. 讓 UserProxyAgent 在方案提供後請求人類確認。
  3. 讓代理生成解決數學問題的 Python 程式碼並執行。

F. 第六週：MCP (Model Context Protocol)

本週轉向基礎且具前瞻性的主題：模型上下文協議 (MCP)，了解 AI 代理與外部世界互動方式的標準化趨勢。

理解 MCP 核心概念、設計目標及其如何為 AI 代理提供標準化方式存取外部工具與數據。若使用 mcp-agent 框架，學習其用法。對應 GitHub 資料夾：6_mcp。

  • 什麼是 MCP？：由 Anthropic 等倡導的開源協議，為 AI 系統與外部軟體、工具、內容儲存庫提供標準化互動介面。
  • MCP 的動機：解決工具整合複雜度和維護成本問題。
  • MCP 的運作方式：代理向 MCP 伺服器發請求，伺服器作中介與實際工具互動。
  • MCP 的優勢：標準化介面、改進的工具存取、可擴展性、社群驅動、身份驗證。
  • mcp-agent 框架：基於 MCP 的 Python 框架，簡化 MCP 代理建構，實現多種有效代理模式 (AugmentedLLM, Router, Orchestrator-Workers 等)。
  1. (若用 mcp-agent) 修改範例，讓代理通過 MCP 連接新工具。
  2. 研究 mcp-agent 的 Router 模式範例，嘗試添加新路由規則。
  3. 思考 MCP 如何標準化先前學習的代理應用中的工具存取。

C. LangGraph：打造複雜代理工作流

LangGraph 以其圖形化的狀態機模型，為處理複雜邏輯、循環和狀態依賴的代理任務提供強大支援。

核心內容：

  • 狀態管理：設計和管理複雜狀態 (State)，更新機制，節點共享與修改狀態。
  • 條件邏輯與循環：使用條件邊 (Conditional Edges) 實現複雜分支邏輯和循環。
  • 人機互動模式：中斷 (interrupt) 和恢復 (resume) 機制的應用 (如人工審批)。

D. AutoGen：自動化多代理對話

AutoGen 在實現自主對話和協作的多 AI 代理方面表現出色，適用於模擬角色間溝通以達成目標的場景。

  • 群組聊天架構：GroupChat 和 GroupChatManager 的使用，設計不同模式群組對話流程。
  • 程式碼執行與安全性：AutoGen 代理執行 LLM 生成程式碼的能力及安全考量。
